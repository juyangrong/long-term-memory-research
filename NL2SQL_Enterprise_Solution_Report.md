## 执行摘要

本报告针对企业级 NL2SQL（Natural Language to SQL）场景，结合现有离线数仓状态（Hive 表、报表、JOB 脚本），提供可落地的技术方案。方案基于三层架构：**业务语义层 + 向量知识库 + RAG 流程**，实现业务人员通过自然语言查询数据。

## 3. 核心组件选型

### 3.1 语义层：dbt MetricFlow

**GitHub:** https://github.com/dbt-labs/metricflow  
**License:** Apache 2.0

**核心能力：**
- 统一定义业务指标（Metrics）
- 支持多数据源（Snowflake, BigQuery, Databricks, Hive）
- 自动处理维度关联和 JOIN 逻辑
- 支持 Open Semantic Interchange (OSI) 标准

**指标定义示例：**
```yaml
metrics:
  - name: gmv
    label: GMV
    type: simple
    type_params:
      measure: gmv_total
    description: "成交总额"
    
  - name: active_users
    label: 活跃用户数
    type: derived
    type_params:
      metrics:
        - daily_active_users
      expr: SUM(daily_active_users)
```

### 3.2 AI 编排：LangChain + LlamaIndex

**LangChain:** 负责整体流程编排（Query → RAG → SQL → Execution）  
**LlamaIndex:** 负责元数据检索和上下文构建

**典型流程：**
1. 用户输入："上周北京地区的订单量是多少？"
2. Query Rewriting：识别时间（上周）、地点（北京）、指标（订单量）
3. RAG 检索：从向量库查找相关表结构、历史 SQL 案例
4. 语义层查询：通过 MetricFlow 生成指标 SQL
5. LLM 优化：使用 DeepSeek-V3/Qwen-Max 优化 SQL
6. 执行并返回结果

### 3.3 向量数据库：Qdrant

**优势：**
- 支持混合检索（BM25 + 向量）
- 高性能，支持大规模元数据
- 开源，可私有化部署

**存储内容：**
- 表结构 schema（embedding）
- 历史 SQL 查询案例
- 业务术语解释文档
- 指标定义元数据

### 3.4 LLM 选型

| 场景 | 推荐模型 | 说明 |
|------|----------|------|
| SQL 生成 | DeepSeek-V3 / Qwen-Max | 代码能力强，支持复杂 JOIN |
| 意图识别 | Qwen2.5-7B (LoRA 微调) | 低成本，可私有部署 |
| 结果解释 | GPT-4 / Claude | 自然语言生成能力强 |

## 8. 结论与建议

### 8.1 技术选型建议

1. **语义层必选**：dbt MetricFlow 是目前最成熟的开源方案
2. **RAG 流程必要**：单纯 LLM 无法理解企业特定表结构和业务逻辑
3. **向量检索关键**：历史 SQL 案例检索可显著提升生成准确率
4. **人工审核不可少**：关键查询需经过人工确认后再执行

### 8.2 实施建议

1. **从小场景切入**：先覆盖 5-10 个核心指标，验证方案可行性
2. **建立反馈闭环**：每次查询结果收集用户反馈，持续优化
3. **文档先行**：完善元数据文档和业务术语解释
4. **安全优先**：实施严格的权限管控和审计机制

### 8.3 预期收益

- **效率提升**：业务查询响应时间从 2-3 天缩短至分钟级
- **人力释放**：数据团队减少 50% 的临时取数需求
- **决策加速**：业务人员可自主探索数据，发现新洞察

## 附录 A：示例查询流程

**用户输入：** "帮我查一下上周北京地区的新增用户数和订单量"

**系统处理流程：**

1. **意图识别** → 识别为"指标查询"类型
2. **实体抽取** → 时间：上周，地区：北京，指标：新增用户数、订单量
3. **RAG 检索** → 查找相关表：ods_crm_user, dwd_order_detail_df
4. **语义层查询** → MetricFlow 生成指标 SQL
5. **LLM 优化** → DeepSeek-V3 优化 JOIN 逻辑
6. **执行返回** → 返回结果并生成自然语言解释

**生成 SQL：**
```sql
SELECT 
    COUNT(DISTINCT u.user_id) AS new_users,
    COUNT(o.order_id) AS orders
FROM ods_crm_user u
LEFT JOIN dwd_order_detail_df o ON u.user_id = o.user_id
WHERE u.city = '北京'
    AND u.create_date BETWEEN '2026-04-14' AND '2026-04-20'
```
