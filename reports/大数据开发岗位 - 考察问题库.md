## 3. 项目经历深挖

### 3.1 项目概况

- 请介绍你最有挑战的项目
- 项目的业务背景是什么？
- 项目的数据规模？（数据量、QPS、集群规模）
- 你在项目中的角色和职责？
- 项目的技术选型？为什么这样选？

### 3.2 技术细节

- 项目架构是怎样的？请画图说明
- 数据流转过程？
- 遇到过什么技术难点？如何解决的？
- 做过哪些性能优化？效果如何？
- 有没有踩过坑？有什么经验教训？

### 3.3 项目验证

**追问技巧：**

1. **细节追问**
   - 具体是怎么实现的？
   - 这个参数为什么这样设置？
   - 当时考虑过其他方案吗？

2. **数据追问**
   - 优化前后的具体数据？
   - 如何验证效果的？
   - 监控指标有哪些？

3. **假设追问**
   - 如果数据量翻 10 倍怎么办？
   - 如果这个组件挂了怎么办？
   - 如果重新做这个项目，你会怎么改进？

### 3.4 项目真实性判断

**可疑信号：**
- 回答模糊，回避细节
- 数据前后矛盾
- 技术细节说不清楚
- 无法解释技术选型原因

**可信信号：**
- 能清晰描述技术细节
- 数据具体且合理
- 能说明权衡取舍
- 有独到的见解和反思

## 4. 系统设计题

### 4.1 实时数据仓库

**题目：** 设计一个实时数据仓库，支持亿级数据量的实时查询和分析

**考察点：**
- 架构设计能力
- 技术选型能力
- 性能优化意识
- 容错机制考虑

**期望答案要点：**
```
数据采集 → Kafka → Flink 实时计算 → 
  ├─> ClickHouse (实时查询)
  ├─> Hudi/Iceberg (数据湖)
  └─> Elasticsearch (检索)

关键考虑:
- 数据一致性保证
- 延迟和吞吐量的权衡
- 扩展性设计
- 监控和告警
```

### 4.2 推荐系统数据平台

**题目：** 设计一个推荐系统的数据平台，支持实时特征计算和离线训练

**考察点：**
- 特征工程理解
- 实时离线一体化
- 数据质量控制

### 4.3 数据中台

**题目：** 如何设计一个数据中台，支持多个业务线的数据需求？

**考察点：**
- 数据治理能力
- 平台化思维
- 多租户设计

## 5. 场景问题

### 5.1 故障排查

**场景 1：** 某天数据平台突然变慢，如何排查？

**期望思路：**
```
1. 确认问题范围（所有任务还是部分？）
2. 查看监控指标（CPU、内存、IO、网络）
3. 检查资源使用（YARN 队列、Kafka 积压）
4. 查看日志（错误日志、慢查询日志）
5. 定位瓶颈（网络？磁盘？计算？）
6. 临时恢复 + 根因分析
```

**场景 2：** Spark 任务频繁失败，如何排查？

**场景 3：** Kafka 消息大量积压，如何处理？

**场景 4：** 数据质量异常，如何定位问题？

### 5.2 技术选型

**场景 1：** 实时计算选 Spark Streaming 还是 Flink？为什么？

**场景 2：** 数据存储选 HBase 还是 ClickHouse？

**场景 3：** 消息队列选 Kafka 还是 Pulsar？

**场景 4：** 数据湖选 Hudi 还是 Iceberg？

### 5.3 数据质量

**场景 1：** 如何保证数据不丢失？

**场景 2：** 如何保证数据一致性？

**场景 3：** 如何发现数据异常？

**场景 4：** 数据对不上，如何排查？
