## ⚙️ 二、配置参数详解

### 2.1 openclaw.json 配置

```json
{
  "agents": {
    "defaults": {
      "memorySearch": {
        "enabled": true,
        "provider": "local",
        "store": {
          "driver": "sqlite",
          "path": "/home/rskuser/.openclaw/memory/main.sqlite",
          "vector": {
            "enabled": true
          }
        },
        "sync": {
          "onSessionStart": true,
          "watch": true,
          "intervalMinutes": 30
        },
        "query": {
          "maxResults": 5,
          "hybrid": {
            "enabled": true,
            "vectorWeight": 0.7,
            "textWeight": 0.3
          }
        },
        "cache": {
          "enabled": true
        }
      }
    }
  }
}
```

### 2.2 参数说明

| 参数 | 值 | 说明 | 优化建议 |
|------|-----|------|----------|
| `enabled` | true | 启用记忆检索 | ✅ 保持启用 |
| `provider` | local | 本地检索引擎 | ✅ 适合私有部署 |
| `driver` | sqlite | SQLite 存储 | ✅ 轻量、无需额外服务 |
| `vector.enabled` | true | 启用向量索引 | ✅ 语义检索核心 |
| `sync.onSessionStart` | true | 会话启动同步 | ✅ 保证记忆新鲜度 |
| `sync.watch` | true | 文件监听 | ✅ 自动检测变更 |
| `sync.intervalMinutes` | 30 | 同步间隔 | ⚠️ 可调整为 15-60min |
| `query.maxResults` | 5 | 最大返回结果数 | ✅ 平衡精度和性能 |
| `hybrid.enabled` | true | 混合检索 | ✅ 最佳实践 |
| `hybrid.vectorWeight` | 0.7 | 向量权重 70% | ✅ 语义优先 |
| `hybrid.textWeight` | 0.3 | 文本权重 30% | ✅ 关键词补充 |
| `cache.enabled` | true | 启用缓存 | ✅ 提升重复查询性能 |

### 2.3 Session Hook 配置

```json
{
  "hooks": {
    "internal": {
      "entries": {
        "session-memory": {
          "enabled": true,
          "messages": 15
        }
      }
    }
  }
}
```

**作用**: 每次会话归档最后 15 条消息到记忆系统，用于后续检索。

## 🔍 四、检索流程

### 4.1 检索触发

每次用户提问前，自动执行以下步骤:

```
1. 用户提问
   ↓
2. memory_search 工具调用 (语义搜索)
   ↓
3. 混合检索引擎执行 (BM25 + 向量)
   ↓
4. 返回 Top-5 相关记忆片段
   ↓
5. 如需精确内容，调用 memory_get 读取指定行
   ↓
6. 将记忆片段注入上下文
   ↓
7. LLM 生成回答
```

### 4.2 检索示例

**用户提问**: "我之前配置过哪些 GitHub 仓库？"

**memory_search 返回**:

## ⚠️ 六、风险与问题

### 6.1 已识别风险

| 风险 | 概率 | 影响 | 缓解措施 | 状态 |
|------|------|------|----------|------|
| 敏感信息泄露 | 低 | 高 | .gitignore + 人工审查 | ✅ 已缓解 |
| 记忆污染 (冲突) | 中 | 中 | 时间戳优先 + 人工整理 | ⚠️ 需优化 |
| 检索性能下降 | 低 | 中 | 缓存 + 定期清理 | ✅ 已缓解 |
| 同步失败 | 低 | 中 | 文件监听 + 会话启动双保险 | ✅ 已缓解 |
| Embedding 模型不可用 | 低 | 高 | 本地 Ollama 部署 | ✅ 已缓解 |

### 6.2 待优化项

1. **记忆冲突检测**: 当前无自动冲突检测机制，依赖人工整理
2. **时间衰减**: 未实现时间衰减算法，旧记忆权重不变
3. **记忆压缩**: 未实现自动摘要压缩，长文本占用空间大
4. **多用户隔离**: 当前为单用户设计，不支持多租户

## 📝 八、最佳实践总结

### 8.1 记忆写入原则

1. **重要性评分**: 严格按 1-5 分评分，决定存储位置
2. **及时性**: 重要信息立即写入，不要依赖"mental notes"
3. **结构化**: 使用 Markdown 格式，便于检索和阅读
4. **定期整理**: 每周/每月审查 MEMORY.md，删除过时信息

### 8.2 检索使用原则

1. **自动触发**: 每次回答前自动搜索相关记忆
2. **精确读取**: 使用 memory_get 只读取需要的行，避免全文件加载
3. **引用标注**: 回答中包含 Source: path#line 便于用户验证
4. **低置信度提示**: 检索结果少或分数低时，明确告知用户

### 8.3 安全保护原则

1. **敏感信息过滤**: API 密钥、密码等不写入记忆
2. **Git 保护**: 严格执行 .gitignore 规则
3. **定期备份**: 使用备份脚本定期备份记忆文件
4. **权限控制**: 生产环境限制记忆文件访问权限
