## 🎯 项目目标

构建企业级 ChatBI 语义层体系，实现：
1. **业务语义统一** - 建立企业级业务术语、指标、维度的标准化定义
2. **NL2SQL 准确率高** - 通过语义层约束，将自然语言查询准确率提升至 90%+
3. **可解释可追溯** - 每个查询都有完整的语义血缘和计算逻辑追溯
4. **自助分析能力** - 业务人员无需 SQL 技能即可进行复杂数据分析

## 📊 核心流程

### 流程 1: 语义层建设（离线）
```
ETL 任务血缘 → 字段级依赖分析 → 业务语义抽取 → 图谱构建 → 语义校验 → 发布
```

### 流程 2: 查询执行（在线）
```
用户自然语言 → NL2DSL 解析 → 语义层校验 → DSL2SQL 转换 → SQL 执行 → 结果返回 → 可视化
```

### 流程 3: 评测迭代（闭环）
```
查询日志 → 抽样标注 → Agent 评测 → 准确率分析 → 问题定位 → 语义层/模型优化
```

## 🔑 关键技术决策

### 1. 语义层存储选型
| 方案 | 优势 | 劣势 | 推荐场景 |
|------|------|------|----------|
| **图数据库 (Neo4j/Nebula)** | 关系查询高效、支持复杂推理 | 学习成本高、运维复杂 | 大型 enterprise、复杂业务关系 |
| **关系型 + JSON** | 技术成熟、易维护 | 复杂关系查询性能弱 | 中小规模、快速启动 |
| **向量数据库** | 语义检索强、支持 RAG | 精确关系表达弱 | 作为辅助检索层 |
| **混合架构** | 兼顾各方面能力 | 架构复杂度高 | ⭐ **推荐** - 图 + 关系 + 向量 |

### 2. NL2DSL 技术方案
| 方案 | 准确率 | 可解释性 | 维护成本 | 推荐度 |
|------|--------|----------|----------|--------|
| 纯 LLM | 60-70% | 低 | 低 | ⭐⭐ |
| LLM+ 规则 | 75-85% | 中 | 中 | ⭐⭐⭐⭐ |
| 微调模型 + 语义层约束 | 85-95% | 高 | 高 | ⭐⭐⭐⭐⭐ |

### 3. DSL 设计原则
- **业务友好** - 接近自然语言的表达
- **可验证** - 语法和语义可静态校验
- **可追溯** - 每个操作都能映射到血缘
- **可扩展** - 支持新指标/维度无缝接入

## 📅 项目里程碑（建议 6 个月周期）

| 阶段 | 时间 | 主要交付物 | 关键指标 |
|------|------|------------|----------|
| **Phase 1: 基础建设** | M1-M2 | 语义层模型设计、Graph 数据库搭建、血缘接入 | 核心业务术语覆盖率 80% |
| **Phase 2: NL2DSL** | M2-M3 | 语义解析引擎、DSL 语法设计、测试集构建 | 简单查询准确率>85% |
| **Phase 3: DSL2SQL** | M3-M4 | 查询生成引擎、执行优化、权限集成 | 查询响应时间<3s |
| **Phase 4: 评测体系** | M4-M5 | 评测 Agent、自动化测试、监控大盘 | 评测覆盖率 100% |
| **Phase 5: 试点上线** | M5-M6 | 业务试点、反馈迭代、文档培训 | 用户满意度>4.0/5.0 |
| **Phase 6: 全面推广** | M6+ | 全业务接入、持续优化 | NL2SQL 准确率>90% |

## ⚠️ 关键风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 语义层建设周期过长 | 高 | 中 | 采用增量建设，优先核心业务 |
| NL2DSL 准确率不达标 | 高 | 中 | 人机协同，保留人工校验环节 |
| 业务部门配合度低 | 中 | 高 | 高层推动 + 激励机制 + 降低使用门槛 |
| 技术选型失误 | 高 | 低 | PoC 验证 + 架构可插拔设计 |
| 数据质量问题 | 中 | 高 | 数据治理前置 + 质量监控 |
