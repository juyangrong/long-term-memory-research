# 技术架构设计

## 1. 架构概述

### 1.1 设计原则

- **语义与数据分离** - 语义层作为独立层，与底层数据存储解耦
- **统一语义出口** - 所有数据消费端（ChatBI、报表、API）共用同一语义层
- **可插拔设计** - 各模块支持独立替换和升级
- **治理先行** - 语义定义需经过审批流程才能发布

### 1.2 系统边界

## 2. 核心模块设计

### 2.1 语义知识图谱模块

**职责：** 存储和管理业务语义知识

**核心实体：**
- **业务术语 (Business Term)** - 企业统一的语言定义
- **指标 (Metric)** - 可量化的业务度量
- **维度 (Dimension)** - 分析的角度和粒度
- **映射关系 (Mapping)** - 语义到物理数据的映射

**技术选型：**
```
图数据库 (Neo4j/Nebula Graph) - 存储语义关系
向量数据库 (Milvus/Weaviate)  - 支持语义检索
关系数据库 (MySQL/PostgreSQL)  - 存储属性数据
```

### 2.2 NL2DSL 语义解析模块

**职责：** 将自然语言查询转换为 DSL

**处理流程：**
```
1. 意图识别 → 2. 实体抽取 → 3. 语义补全 → 4. DSL 生成 → 5. 语义校验
```

**技术实现：**
- 基于 LLM 的语义理解
- 基于规则的后处理
- 基于语义层的约束校验

### 2.3 DSL2SQL 查询生成模块

**职责：** 将 DSL 转换为可执行的 SQL

**核心能力：**
- DSL 语法解析
- 语义映射查找
- SQL 生成优化
- 权限校验注入

### 2.4 评测与迭代模块

**职责：** 持续监控和优化语义层质量

**关键指标：**
- NL2SQL 准确率
- 查询响应时间
- 语义覆盖率
- 用户满意度

## 4. API 设计

### 4.1 语义检索 API

```yaml
POST /api/v1/semantic/search
Body:
  query: string        # 搜索关键词
  type: "term|metric|dimension"  # 实体类型
  limit: number        # 返回数量
  
Response:
  items: [{
    id: string
    name: string
    type: string
    description: string
    score: number
  }]
```

### 4.2 语义解析 API

```yaml
POST /api/v1/semantic/parse
Body:
  natural_language: string  # 用户自然语言
  context: object           # 上下文信息
  
Response:
  dsl: string               # 生成的 DSL
  entities: [{              # 识别的实体
    type: string
    name: string
    confidence: number
  }]
  confidence: number        # 解析置信度
```

### 4.3 DSL 执行 API

```yaml
POST /api/v1/dsl/execute
Body:
  dsl: string               # DSL 语句
  
Response:
  sql: string               # 生成的 SQL
  data: array               # 查询结果
  metadata: object          # 元数据信息
```

## 6. 安全设计

### 6.1 认证授权

- **API 认证** - JWT Token 认证
- **权限控制** - 基于 RBAC 的细粒度权限
- **数据权限** - 与 DataCloud 权限体系打通

### 6.2 数据安全

- **传输加密** - HTTPS/TLS
- **存储加密** - 敏感字段加密存储
- **审计日志** - 所有操作留痕

### 6.3 语义安全

- **语义校验** - 防止恶意查询
- **查询限制** - 防止资源滥用
- **敏感词过滤** - 合规性检查
