# 05-测试评估体系方案

## 📖 概述

本文档描述 ChatBI 语义层的测试评估体系设计，包括评测 Agent 架构、测试集构建、准确率评估和持续迭代机制。

---

## 🎯 建设目标

1. **自动化评测** - 90% 以上测试用例自动化执行
2. **全面覆盖** - 覆盖核心业务场景 95%+
3. **快速反馈** - 评测结果 T+1 输出
4. **持续优化** - 建立闭环迭代机制

## 📊 评测维度

### 1. 功能准确性

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 意图识别准确率 | 正确识别用户意图的比例 | > 90% |
| 实体抽取 F1 值 | 实体识别的精确率和召回率 | > 85% |
| 语义链接准确率 | 实体链接到正确语义节点的比例 | > 88% |
| DSL 生成正确率 | 生成合法 DSL 的比例 | > 95% |
| SQL 生成准确率 | 生成可执行且正确 SQL 的比例 | > 99% |
| 查询结果准确率 | 最终结果与预期一致的比例 | > 90% |

### 2. 性能指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 端到端延迟 | 从用户输入到结果返回的总时间 | P95 < 3s |
| 语义解析耗时 | NL2DSL 转换时间 | P95 < 500ms |
| SQL 生成耗时 | DSL2SQL 转换时间 | P95 < 100ms |
| 查询执行耗时 | SQL 执行时间 | P95 < 2s |
| 系统吞吐量 | 单位时间处理查询数 | > 1000 QPS |

### 3. 用户体验

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 用户满意度 | 用户对结果的评分 | > 4.0/5.0 |
| 任务完成率 | 用户成功完成查询的比例 | > 85% |
| 平均交互轮次 | 完成查询所需的平均对话轮数 | < 2.5 |
| 错误恢复率 | 用户从错误中恢复的比例 | > 70% |

### 4. 系统稳定性

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 系统可用性 | 系统正常运行时间比例 | > 99.9% |
| 错误率 | 请求失败的比例 | < 0.1% |
| 平均故障间隔 | 两次故障之间的平均时间 | > 720h |

## 🤖 评测 Agent 设计

### 1. 用例生成 Agent

**职责：**
- 基于业务场景生成测试用例
- 自动扩充边界条件测试
- 生成对抗性测试用例

**工作流程：**
```
业务场景库
  ↓
模板填充
  ↓
变体生成
  ↓
用例审核
  ↓
测试用例库
```

**用例格式：**
```json
{
  "case_id": "CASE_001",
  "category": "对比查询",
  "business_domain": "交易域",
  "natural_language": "查看华东和华南地区上季度的销售额对比",
  "expected_intent": "compare",
  "expected_entities": [
    {"type": "", "value": "华东"},
    {"type": "", "value": "华南"},
    {"type": "metric", "value": "销售额"},
    {"type":", "value": "上季度"}
  ],
  "expected_dsl": "SELECT 指标 (销售额) WHERE 维度 (地区，IN，[华东，华南]) AND 维度 (时间，=，上季度) GROUP BY 维度 (地区)",
  "expected_sql": "SELECT region, SUM(sales_amount) FROM fact_sales WHERE region IN ('华东', '华南') AND quarter = 'Q1' GROUP BY region",
  "difficulty": "medium",
  "tags": ["region", "time", "comparison"]
}
```

### 2. 执行引擎 Agent

**职责：**
- 执行测试用例
- 捕获中间结果
- 记录性能指标
- 处理异常情况

**执行流程：**
```
测试用例
  ↓
NL 输入 → 意图识别 → 实体抽取 → 语义链接 → DSL 生成 → SQL 生成 → SQL 执行
  ↓          ↓           ↓           ↓           ↓          ↓          ↓
      记录各阶段结果和耗时
  ↓
结果汇总
```

### 3. 结果比对 Agent

**职责：**
- 比对实际结果与预期结果
- 计算准确率指标
- 生成差异报告
- 定位问题根因

**比对策略：**
1. **精确匹配** - 结果完全一致
2. **语义匹配** - 结果语义等价
3. **模糊匹配** - 结果在可接受误差范围内

**差异分析：**
```
预期结果 vs 实际结果
  ↓
差异检测
  ↓
根因分析：
  - 意图识别错误？
  - 实体抽取错误？
  - 语义链接错误？
  - DSL 生成错误？
  - SQL 生成错误？
  - 数据问题？
  ↓
问题分类 + 修复建议
```

## 📊 质量门禁

### 1. 上线标准

| 指标 | 阈值 | 说明 |
|------|------|------|
| 功能测试通过率 | 100% | 所有用例必须通过 |
| 核心场景准确率 | > 90% | 高频查询场景 |
| 性能指标达标率 | 100% | 延迟/吞吐量要求 |
| 严重 Bug 数 | 0 | P0/P1 级别必须修复 |
| 代码覆盖率 | > 80% | 单元测试覆盖 |

### 2. 降级标准

| 场景 | 触发条件 | 降级策略 |
|------|----------|----------|
| 准确率下降 | 核心指标 < 85% | 切换至规则模式 |
| 性能恶化 | P95 延迟 > 5s | 限流 + 降级 |
| 系统故障 | 错误率 > 1% | 熔断 + 告警 |
| 数据异常 | 结果偏差 > 10% | 暂停服务 + 排查 |

## 🔄 持续迭代机制

### 1. 反馈闭环

```
用户查询 → 系统处理 → 结果返回
              ↓
         用户反馈 (点赞/点踩/修正)
              ↓
         反馈收集 + 标注
              ↓
         问题分析 + 归类
              ↓
         模型/规则优化
              ↓
         回归测试验证
              ↓
         版本发布
```

### 2. 数据飞轮

**正向循环：**
- 更多用户 → 更多数据 → 更好模型 → 更好体验 → 更多用户

**关键指标：**
- 日活跃用户数
- 查询量增长率
- 用户留存率
- 准确率提升曲线

### 3. A/B 测试

**测试策略：**
- 小流量灰度（1% → 5% → 20% → 50% → 100%）
- 多版本并行
- 统计显著性检验

**评估维度：**
- 准确率对比
- 性能对比
- 用户满意度对比
