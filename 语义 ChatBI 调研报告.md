## 🔍 调研背景

### 业务需求
- 降低数据分析门槛，让业务人员通过自然语言查询数据
- 提升数据使用效率，减少重复取数需求
- 统一数据语义，避免指标口径不一致问题

### 技术驱动
- LLM 能力成熟，支持复杂的语义理解和 SQL 生成
- 企业数据平台完善，具备统一数据服务基础
- 用户对智能化交互体验期望提升

## 💬 ChatBI 核心概念

### 定义
ChatBI = Chat + BI，通过自然语言对话实现数据查询、分析和可视化的智能系统。

### 核心价值
| 维度 | 传统 BI | ChatBI |
|------|---------|--------|
| 使用门槛 | 需要 SQL/工具培训 | 自然语言对话 |
| 响应速度 | 小时/天级 | 秒级 |
| 灵活性 | 固定报表 | 任意问题 |
| 覆盖人群 | 数据分析师 | 全员 |

### 典型场景
1. **即时查询**："上周华东区销售额是多少？"
2. **趋势分析**："近 3 个月用户增长趋势如何？"
3. **对比分析**："A 产品和 B 产品的转化率对比"
4. **归因分析**："为什么本周订单量下降了？"
5. **预测建议**："下季度库存应该准备多少？"

## 🏗️ 语义层架构

### 四层语义模型

```
┌─────────────────────────────────────┐
│         应用层 (Application)         │
│  Chat 界面 / 报表 / 预警 / 推荐      │
└─────────────────────────────────────┘
              ↓ ↑
┌─────────────────────────────────────┐
│         语义层 (Semantic)            │
│  术语 / 指标 / 维度 / 修饰词 / 关系  │
└─────────────────────────────────────┘
              ↓ ↑
┌─────────────────────────────────────┐
│         逻辑层 (Logical)             │
│  SQL 生成 / 查询优化 / 执行计划       │
└─────────────────────────────────────┘
              ↓ ↑
┌─────────────────────────────────────┐
│         物理层 (Physical)            │
│  Hive / StarRocks / MySQL / API     │
└─────────────────────────────────────┘
```

### 语义元素定义

#### 1. 术语 (Term)
业务概念的标准化表达
- 示例：`GMV`、`DAU`、`转化率`
- 属性：名称、定义、业务域、责任人

#### 2. 指标 (Metric)
可量化的业务度量
- 示例：`日销售额`、`月活跃用户数`
- 属性：名称、计算公式、统计周期、维度

#### 3. 维度 (Dimension)
分析的角度和粒度
- 示例：`时间`、`地区`、`产品线`
- 属性：名称、层级、取值范围

#### 4. 修饰词 (Modifier)
限定条件
- 示例：`线上`、`新客`、`促销期`
- 属性：名称、适用指标、逻辑表达式

## ⚙️ 技术实现方案

### 整体架构

```
用户提问 → 语义理解 → 查询生成 → 执行优化 → 结果呈现
              ↓           ↓           ↓
          语义层      逻辑层      物理层
```

### 关键模块

#### 1. 语义理解 (NLU)
- **意图识别**：分类查询类型（即时查询/趋势分析/对比/归因）
- **实体抽取**：识别指标、维度、时间范围、筛选条件
- **语义解析**：将自然语言映射到语义层元素

#### 2. 查询生成 (NL2SQL)
- **模板匹配**：基于预定义 SQL 模板填充参数
- **LLM 生成**：使用大模型生成 SQL（需验证准确性）
- **混合方案**：模板为主，LLM 兜底

#### 3. 执行优化
- **查询改写**：优化 SQL 性能
- **缓存策略**：热点查询结果缓存
- **限流降级**：保护底层数据源

#### 4. 结果呈现
- **智能图表**：根据数据类型自动选择可视化方式
- **文字解读**：LLM 生成分析结论
- **下钻能力**：支持进一步细化查询

### 技术选型建议

| 模块 | 方案 A | 方案 B | 推荐 |
|------|--------|--------|------|
| NLU | 规则 + 分类模型 | LLM Few-shot | 方案 B（灵活性高） |
| NL2SQL | LLM 直接生成 | 语义层映射 | 方案 B（准确率高） |
| 执行引擎 | 自研 | DataCloud/Zeus | 复用现有平台 |
| 前端 | 飞书机器人 | 独立 Web 应用 | 飞书（集成度高） |

## 🎯 落地建议

### 阶段一：MVP（1-2 个月）
- ✅ 选择 3-5 个高频查询场景
- ✅ 构建核心指标语义层（50+ 指标）
- ✅ 实现模板化 NL2SQL
- ✅ 飞书机器人集成

### 阶段二：扩展（3-6 个月）
- ✅ 覆盖 80% 常规分析需求
- ✅ 引入 LLM 提升语义理解
- ✅ 支持多轮对话和上下文
- ✅ 添加智能图表推荐

### 阶段三：深化（6-12 个月）
- ✅ 归因分析和预测能力
- ✅ 自助语义建模工具
- ✅ 全公司推广培训
- ✅ 建立运营和反馈机制

### 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 语义理解准确率低 | 高 | 人工审核 + 持续训练 |
| SQL 生成错误 | 高 | 严格验证 + 限流保护 |
| 用户期望过高 | 中 | 明确边界 + 教育引导 |
| 数据质量差 | 高 | 先治理数据再上线 |
| 使用率低 | 中 | 场景驱动 + 激励机制 |

### C. FAQ

**Q: ChatBI 会取代数据分析师吗？**
A: 不会。ChatBI 处理标准化查询，分析师专注于深度分析和策略建议。

**Q: 如何保证数据准确性？**
A: 三层验证：语义层校验 + SQL 审核 + 结果比对。

**Q: 敏感数据如何保护？**
A: 行级权限控制 + 查询审计 + 脱敏展示。
